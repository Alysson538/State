State Pattern

Alterando o comportamento de um objeto conforme seu estado interno.

O Problema

Como representar diferentes comportamentos de um objeto dependendo do seu estado?

Exemplo

Cada pedido de compra pode assumir múltiplos estados:

  • Novo
  • Pago
  • Enviado
  • Finalizado
  • Cancelado

Código Sem o Pattern


class Pedido {
  String estado;

  void proximoEstado() {
    if (estado.equals("Novo")) {
      estado = "Pago";
    } else if (estado.equals("Pago")) {
      estado = "Enviado";
    } else if (estado.equals("Enviado")) {
      estado = "Finalizado";
    }
  }
}

O Problema Resumido

  • Mudanças quebram o código
  • Regras espalhadas
  • Viola Open/Closed

Intenção do State Pattern

O State permite que um objeto altere seu comportamento quando seu estado interno muda, parecendo mudar sua própria classe.

  • Cada estado vira uma classe separada
  • O comportamento muda dinamicamente
  • Elimina if/else completamente

Como Funciona (Estrutura)

  • Context: mantém referência ao estado atual e delega ações
  • State (Interface): define operações comuns aos estados
  • Concrete States: implementam comportamento específico
  • Cada estado pode alterar o estado do contexto

Exemplo Prático em Java


interface EstadoPedido {
  void proximoEstado(Pedido pedido);
}

class Novo implements EstadoPedido {
  public void proximoEstado(Pedido pedido) {
    pedido.setEstado(new Pago());
  }
}

class Pago implements EstadoPedido {
  public void proximoEstado(Pedido pedido) {
    pedido.setEstado(new Enviado());
  }
}
        

class Pedido {
  private EstadoPedido estado;

  public Pedido() {
    estado = new Novo();
  }

  public void proximoEstado() {
    estado.proximoEstado(this);
  }
}

// Uso:
Pedido pedido = new Pedido();
pedido.proximoEstado(); // Novo -> Pago
pedido.proximoEstado(); // Pago -> Enviado

Aplicações no Mundo Real

  • E-commerce: ciclo de vida de pedidos com estados bem definidos
  • Jogos: personagem parado, correndo ou pulando
  • Apps: modo online/offline, APIs de pagamento, microserviços

State vs Strategy

  • Strategy: troca comportamento manualmente
  • State: troca automaticamente conforme o estado interno

Vantagens e Desvantagens

  • Vantagens: remove if/else, código mais organizado, fácil adicionar novos estados
  • Desvantagens: muitas classes para estados simples, mais complexo para iniciantes

Conclusão

Use o State Pattern quando:

  • O comportamento depende do estado
  • Há muitos if/else
  • Mudanças são frequentes
